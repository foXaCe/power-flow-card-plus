## Findings considered and rejected

(Recorded so they are not re-audited next run. All verified by reading the cited code.)

- **"XSS via `style_card_content` / `style_ha_card` / arrow `color`"** (reported by
  the security pass): NOT a vulnerability. (1) These are Lit attribute bindings
  (`style=${...}` inside `html`/`svg` templates) — Lit sets them via the DOM, not
  string concatenation, so a quote in the value cannot break out into a new
  attribute / `onclick`. There is zero `innerHTML`/`unsafeHTML`/`eval` in `src/`.
  (2) These config options exist by design to let users style their own card; the
  only actor who can set them is whoever edits the Lovelace dashboard, who already
  has full control (can add iframe/webpage cards, custom JS resources). No
  privilege boundary is crossed.
- **"Division by zero in flow-animation `setCurrentTime`"** (`power-flow-card-plus.ts:~700`):
  already guarded — the condition `... && this.previousDur[flowName] && ...` is
  falsy when the value is `0` or `undefined`, so the division never runs with a
  zero denominator.
- **"`memoize-one` caches stale circle centers after drag"** (`components/flows/index.ts:50,73`):
  non-issue. `memoizeOne` holds one entry; the function is called with a different
  `circleClass` arg for each circle within a render, so the cache is constantly
  evicted and `getBoundingClientRect` is read live each time — no staleness, and
  effectively little memoization benefit (not a correctness bug).
- **"localStorage prototype pollution / unvalidated parse"** (`power-flow-card-plus.ts:~1083`):
  low/no real risk — localStorage is same-origin and device-local (not
  attacker-controlled), and plain object spread of position data does not pollute
  `Object.prototype`. Defensive validation is nice-to-have, not worth a plan.
- **Negative-zero edge cases** in tolerance/allocation: theoretical; no observable
  user-facing effect (`Math.max(..., 0)` and display rounding absorb it).
